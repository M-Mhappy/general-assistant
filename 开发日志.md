### 2026-06-14: Python daemon 编码导致 JSON 协议通信失败

**现象**：Node.js 主进程调用 `executor.execute()` 时间歇性超时，特别是在 code 或 cwd 包含中文时。`StreamProxy` 产生无限递归，输出 JSON 消息被当成原始文本再被 JSON 编码，指数级膨胀。

**原因**：
1. Windows 上 Python 默认 stdout/stderr 编码为 GBK，中文路径和代码输出产生非 UTF-8 字节流。Node.js 侧 `chunk.toString()` 默认 UTF-8 解码，遇到 GBK 字节时产生乱码或解析失败，JSON 行分割错乱导致永远收不到 `executeResult`。
2. `send_message()` 使用 `sys.stdout` 发送协议消息，但在代码执行期间 `sys.stdout` 已被替换为 `StreamProxy`，导致 `send_message → StreamProxy.write → send_message` 无限递归。

**方案**：
1. Python daemon 启动时显式调用 `sys.stdout.reconfigure(encoding='utf-8')` 和 `sys.stderr.reconfigure(encoding='utf-8')`，并在 Node.js spawn 时传入 `PYTHONIOENCODING=utf-8` 环境变量。
2. 引入 `_REAL_STDOUT` 全局变量，在模块加载时保存真实 stdout 引用。`send_message()` 始终通过 `_REAL_STDOUT` 写入，避开 StreamProxy。
3. `StreamProxy` 移除对 `self.original` 的直接写入，只通过 `send_message()` 发送流式输出 + 写入 `StringIO` 捕获缓冲区。

**教训**：跨进程 JSON-line 通信时，必须在双方显式约定字符编码，不能依赖系统默认。Python 的 `sys.stdout` 替换模式（用于捕获输出）必须与自定义协议通信通道物理隔离——用不同的文件描述符或在替换前保存原始引用。

**关联**：`python-executor/python-daemon.py`, `modules/python-executor.js`

### 2026-06-14: 表达式求值导致代码重复执行

**现象**：多行 Python 代码执行时，最后一行的输出重复出现（如 `print("line2")` 输出两次）。

**原因**：`CodeExecutor.execute()` 先 `exec(compiled, globals)` 执行全部代码，然后尝试将最后一行作为表达式 `eval()`。对于 `print("line2")` 这类函数调用，Python AST 将其解析为合法的 `Expression` 节点（Call），导致 `eval()` 再次执行 `print()`。

**方案**：引入 `_ast_contains_call_or_assignment()` 辅助函数，遍历最后一行代码的 AST，若包含 `Call` 或 `NamedExpr` 节点则跳过 eval。纯值表达式（如 `1+2`、`x`、`[i*2 for i in range(5)]`）不受影响。

**教训**：`exec()` + `eval()` 组合模式必须用 AST 判断最后一行的节点类型，不能仅靠字符串前缀检查（如 `startswith("print")`），因为函数调用可以有无数种写法。

**关联**：`python-executor/python-daemon.py`
